# DatsSol — спецификация стратегии для Codex

Ниже не “красивые идеи”, а рабочая доктрина под алгоритм. Главная мысль: в DatsSol выигрывает не самый агрессивный и не самый мирный бот, а тот, кто лучше конвертирует **каждое действие в очки с минимальным риском потери сети**.

---

## 0) Что реально решает игру

### 0.1. Это игра про темп сети, а не про отдельные дуэли
- Каждая плантация живёт ограниченное время: при `TS=5` клетка закрывается за 20 ходов.
- Значит, если бот не завершает новые стройки примерно с тем же темпом, с которым старые плантации исчезают, сеть начинает схлопываться.
- Базовая метрика: `completion_rate_20 = число завершённых плантаций за последние 20 ходов / 20`.
- Нужно держать `completion_rate_20 >= expiration_rate_20`. На практике комфортно держать чуть выше.

### 0.2. Тонкая змейка — мусорная архитектура
- Управление держится на ортогональной связности от ЦУ.
- Любой мост/перемычка — критическая точка. Если её срезают, дальняя часть сети становится изолированной, перестаёт приносить очки и деградирует.
- Поэтому выгоднее **двухполосная/компактная лента** с резервными связями, а не длинный одинарный хвост.

### 0.3. Клетки x%7==0 и y%7==0 — приятный бонус, но не религия
- Усиленная клетка даёт +500 к общей ценности клетки.
- Но такие клетки редки: 1 из 49.
- Средняя прибавка по карте — около **+10.2 очка на клетку в среднем**, то есть чуть больше 1% к средней стоимости клетки.
- Вывод: **предпочитать усиленные клетки как tie-break — да. Строить всю макростратегию только вокруг них — нет.**

### 0.4. Бобры — жирнее большинства обычных решений
- Логово стоит `10 x ценность клетки`, то есть 10k или 15k очков, плюс после убийства клетку можно терраформить.
- Это огромный источник рывков по раундовым очкам.
- Но нельзя “честно долго пилить” бобра, если кто-то может украсть last-hit большим burst-уроном.

### 0.5. Чип-урон без плана добивания — благотворительность
- Ремонт идёт **до** диверсий.
- Если ты наносишь 5–10 урона в ход по поддерживаемой цели без kill-plan, ты просто даришь сопернику повод тратить 1 действие на ремонт и продолжаешь кормить третьего игрока, который спокойно фармит карту.

### 0.6. Враг в API частично “анонимный”
- В `enemy[]` нет `ownerId` и нет `isMain`.
- Значит, нельзя строить стратегию на “вот это точно ЦУ конкретного топа”.
- Работать надо по **видимым кластерам врага**, мостам, критическим узлам и локальному превосходству.

## 1) Архитектура бота

Нужна не пачка жёстких if/else, а **utility-based planner + state machine режимов**.

### 1.1. Слои
1. **Парсер мира**
   - читает `/api/arena`
   - обновляет карту, свои плантации, enemy-кластеры, стройки, бобров, прогнозы
2. **Трекер памяти**
   - помнит прогресс видимых строек по ходам
   - помнит появление/исчезновение enemy-клеток
   - считает собственные fail-streak по командам
3. **Оценка состояния**
   - риск ЦУ
   - хрупкость сети
   - темп завершения строек
   - локальное превосходство по фронтам
   - EV бобров/диверсий/строек
4. **Выбор режима**
   - economy / raid / defense / contested / rebase
5. **Генератор задач**
   - build / finish build / repair / sabotage / beaver / relocate main
6. **Аллокатор действий**
   - выбирает пары `author -> output -> target`
   - учитывает штраф загрузки выходной плантации
7. **Слой ручных оверраидов**
   - не командует юнитами напрямую, а меняет веса и ограничения

## 2) Ключевые метрики, которые нужно считать каждый ход

### 2.1. По своей сети
- `connected_count`
- `isolated_count`
- `construction_count`
- `active_limit_headroom = settlement_limit - (plantations + constructions)`
- `hq_remaining_turns = ceil((100 - terraform_progress_under_hq)/TS)`
- `safe_hq_moves_count` — сколько есть безопасных adjacent-кандидатов для переноса ЦУ
- `completion_rate_20`
- `expiration_rate_20`
- `idle_author_ratio`
- `output_congestion_ratio`

### 2.2. По хрупкости графа
На графе из своих плантаций каждый ход прогонять Tarjan / articulation points.

Для каждой своей плантации `v` считать:
- `criticality(v)` = сколько будущей ценности сети теряется, если убрать `v`
- `bridge_subtree_value(v)` = сумма remaining value зависимых веток

Для видимого enemy-кластера считать то же самое, но с коэффициентом доверия (из-за тумана войны).

### 2.3. По фронтам
Для каждого локального театра (сектор / frontier cluster):
- `our_local_damage_cap`
- `our_local_repair_cap`
- `enemy_local_damage_cap_visible`
- `enemy_local_support_count_visible`
- `beaver_ev`
- `sector_space_quality`
- `storm_risk`
- `mountain_choke`

### 2.4. По режиму конца раунда
- `turns_left = round_end - turnNo`
- чем меньше `turns_left`, тем меньше вес “долгой инфраструктуры” и больше вес немедленных очков / denial / secure-finish

## 3) Генератор задач: базовые типы

### 3.1. BuildTask
Кандидат — пустая клетка в допустимой геометрии.

Скоринг:
- `cell_value` = 1000/1500
- `score_horizon` = сколько ходов клетка ещё успеет работать до конца раунда
- `support_multiplicity` = сколько разных выходных плантаций могут эффективно поддерживать стройку
- `connectivity_bonus` = создаёт ли клетка цикл / резервный путь / хороший следующий anchor для ЦУ
- `frontier_bonus` = сколько свободных клеток и обзора открывает
- `risk_penalty` = бобры / враг / буря / choke
- `manual_bias`

Итоговая идея:
`build_score = immediate_value + frontier_bonus + connectivity_bonus + support_bonus - risk_penalty`

### 3.2. FinishConstructionTask
Очень важный тип задачи. Часто выгоднее **закрыть уже начатую стройку**, чем открыть новую.

Приоритет сильно растёт если:
- стройка может завершиться **в этот ход**
- ход с землетрясением (`turnsUntil=0`) и завершение даст иммунитет
- стройка на critical cell / contested cell

### 3.3. RepairTask
Ремонтировать не всё подряд.

Приоритет ремонта:
- ЦУ
- articulation/bridge nodes
- узлы, от которых зависит следующий перенос ЦУ
- узлы рядом с бобрами/врагом/бурей
- фронтовые кластеры с высоким EV

Низкий приоритет ремонта:
- листы, у которых клетка уже почти добита и которые скоро исчезнут сами
- дальняя expendable-периферия

### 3.4. SabotageTask
Атаковать только если есть одно из:
- kill in 1–2 turns
- разрушение видимого articulation point
- цель под бобром/бурей/quake-finish
- contest for critical zone
- подтверждённый высокий deny value

Никогда не тратить ход на “символическую пощёчину” полной и хорошо поддерживаемой цели.

### 3.5. BeaverTask
Атаковать логово только если:
- суммарный **чистый** урон > 5 в ход (из-за регена)
- либо есть план решающего burst last-hit
- либо зона uncontested

### 3.6. RelocateMainTask
Перенос ЦУ — не опция, а постоянная часть маршрута.

Кандидат на новое ЦУ:
- ортогонально смежен с текущим ЦУ
- connected
- проживёт ещё минимум несколько ходов
- не под немедленным kill-risk
- имеет высокий degree / хорошие пути продолжения

## 4) Аллокатор действий и штрафы за общий output

Все команды через одну выходную плантацию режутся по эффективности: 5, 4, 3, 2, 1 ...

### 4.1. Практические правила
- в economy-режиме обычный лимит — **не больше 2–3 команд через один output**
- 4-я и дальше команды через один output допустимы только для:
  - secure finish стройки
  - secure kill
  - last-hit бобра
  - emergency defense

### 4.2. Критическая микрофишка
Если несколько авторов могут бить одну цель через **разные** выходные плантации, это почти всегда лучше, чем всем стрелять через один output.

Следствие:
- компактные кластеры лучше длинных лент
- frontier-cells нужно выбирать так, чтобы их можно было поддерживать **несколькими distinct outputs**

### 4.3. Рекомендуемый greedy allocator
1. Сгенерировать задачи с base-score.
2. Для каждой задачи найти лучшие пары `(author, output)`.
3. Идти по задачам сверху вниз.
4. Для каждой пары пересчитывать marginal utility с учётом `output_load[output]`.
5. Назначать действие, если utility > 0 и автор ещё не использован.

## 5) Стратегия №1 — базовая победная: Economy / Rolling Carpet

Это дефолт. Всё остальное — временные отклонения.

### 5.1. Главная идея
Построить **устойчивую, не хрупкую, высокотемповую сеть**, которая:
- стабильно держит completion tempo
- не разваливается от одного bridge-kill
- не дарит врагу лёгкие точки диверсий
- opportunistically добирает жирные цели (бобры, слабые узлы, contested cells)

### 5.2. Пространственная форма
Не одиночная змейка, а:
- двухполосная лента,
- местами расширяющаяся до 2x2 / 2x3 узлов,
- с резервными связями,
- с заранее подготовленными adjacent-anchor для переноса ЦУ.

### 5.3. Opening
1. Сразу строить первую adjacent-плантацию для будущего переноса ЦУ.
2. После появления первой опоры — не тянуть один хвост, а делать **вторую полосу** / компактный блок.
3. К 20-му ходу у ЦУ должно быть:
   - минимум 1 безопасная adjacent-замена,
   - желательно 2 варианта,
   - лучше уже начатая следующая глубина сети.

### 5.4. Макроправила economy mode
- 60–75% действий — build / finish build
- 10–20% — repair critical nodes
- 0–15% — beaver opportunities
- 0–10% — opportunistic sabotage

### 5.5. Правила по стройкам
- Держать мало, но поддерживаемых строек: обычно **2–5** одновременно.
- Не открывать стройку, если нельзя гарантировать ей поддержку на следующем ходу.
- Не делать стройки “впрок” по 5–10% и потом бросать: DS съест tempo.
- Предпочитать клетки с высоким `support_multiplicity`.
- Предпочитать клетки, создающие цикл/backup-link.

### 5.6. Перенос ЦУ
Текущий HQ должен жить по правилу:
- если до закрытия клетки `<= 3 хода`, перенос готовится немедленно
- если рядом угроза диверсии/бобра/бури, перенос ускоряется
- всегда держать **минимум 1, лучше 2** adjacent-anchor под next HQ

### 5.7. Экономические приоритеты ход за ходом
Порядок:
1. Спасти ЦУ / критический мост
2. Достроить стройки, которые можно закончить прямо сейчас
3. Построить/укрепить следующий участок backbone
4. Забрать выгодного бобра, если secure
5. Починить critical nodes
6. Только потом — короткие рейды по врагу

### 5.8. Когда economy mode временно включает агрессию
Даже в экономике можно и нужно бить врага, если:
- kill за 1–2 хода
- видно articulation point
- враг залез под бобра/бурю
- идёт contest за critical cell
- до конца раунда мало ходов и быстрый kill важнее новой стройки

### 5.9. Что нельзя делать в economy mode
- не строить культ из boosted cells
- не лезть в долгую равную драку
- не держать 7 полустройек на соплях
- не тратить 4+ команд через один output без крайней причины
- не подставлять HQ под лимит плантаций

## 6) Стратегия №2 — Opportunistic Raid (против слабого соседнего кластера)

Это не “идём воевать”, а **короткое ограбление окна слабости**.

### 6.1. Когда включать
- видимый enemy-cluster слабее локально
- low HP цели
- мало видимых support-узлов врага
- есть articulation target
- враг под бобром или бурей
- мы можем убить/перерезать за 1–3 хода

### 6.2. Цели по приоритету
1. Видимый articulation / bridge
2. Low-HP plant с плохой поддержкой
3. Вражеская стройка на важной клетке
4. Вражеский узел в beaver radius
5. Near-complete target (если тест подтвердит выгодный kill-bonus)

### 6.3. Правила рейда
- commit window: 2–4 хода, не больше без пересмотра
- использовать разные output для burst
- не снимать защиту с собственного HQ и backbone
- если за 2 хода enemy repair поглощает рейд — выйти из режима
- после kill/cut сразу вернуться в economy

### 6.4. Смысл
Идея не в том, чтобы “убить игрока”. Идея — **дёшево снять с него tempo, ресурсы и локальную сеть**, пока мы продолжаем фармить карту.

## 7) Стратегия №3 — Fortress / Counterbattery (если активно атакуют нас)

### 7.1. Когда включать
- 2+ хода подряд идёт входящий урон в один сектор
- HQ в риске
- наш critical bridge под давлением
- visible enemy local damage >= наш local repair/defense capacity

### 7.2. Главная логика
Не лечить всё подряд, а **сохранить управляемость сети**.

### 7.3. Приоритеты
1. ЦУ
2. Узлы, которые обеспечивают next HQ move
3. Articulation nodes и bridge nodes
4. Постройка backup-link вокруг threatened bridge
5. Контрсаботаж по ближайшим вражеским output/bridge, если это снижает входящий урон лучше ремонта
6. Отпустить expendable-листья

### 7.4. Важные правила
- Не чинить то, что уже почти отработало и вот-вот исчезнет, если это не critical node.
- Если прогноз quake = 0, чинить только те узлы, которые обязаны пережить удар.
- Переносить HQ заранее, а не “в последний вдох”.
- Если есть возможность построить вторую связь, это часто лучше, чем бесконечно ремонтировать один мост.

### 7.5. Контрсаботаж
Если враг давит из маленького числа ближайших узлов, выгодно:
- не только repair,
- а выносить его output-узлы или его articulation,
- чтобы следующее входящее давление просто пропало.

## 8) Стратегия №4 — Contested Frontier / Anti-Leader Denial

Это режим спорной зоны: рядом сильный или равный враг, и у вас общий лакомый сектор / choke / beaver.

### 8.1. Когда включать
- видимый сильный enemy-cluster рядом
- борьба за один и тот же фронтир
- общий бобёр / общий choke / критическая стройка
- прямая “честная война” будет невыгодной обоим

### 8.2. Что делать
- contest enemy constructions на ключевых клетках
- сбивать bridges только если kill быстрый
- красть beaver last-hit burst-ом
- строить компактный локальный плацдарм из иммунных свежих плантаций
- избегать долгого симметричного обмена диверсиями

### 8.3. Тактика contested construction
Если враг строит critical cell:
- либо достроить раньше него,
- либо синхронно довести до completion в тот же ход и обнулить стройку всем,
- либо заставить его продолжать вливать туда tempo вместо расширения.

### 8.4. Тактика contested beaver
Никогда не “мягчить” бобра для соперника, если не уверен в secure finish.
- либо забираешь uncontested,
- либо приходишь на решающий burst,
- либо не трогаешь.

### 8.5. Когда выйти
Если contested-zone перестала окупаться по EV за 5–10 ходов, режим выключается и сеть смещается в cleaner sector.

## 9) Стратегия №5 — Rebase / Escape / Recovery

Если локальный театр плохой, не надо умирать героически. Героизм в логах очков не даёт.

### 9.1. Когда включать
- рядом сильный враг и драка невыгодна
- сектор сжат горами/бобрами/врагом
- не хватает пространства для здорового ribbon-growth
- после респавна
- слишком много времени уходит на defense без возврата в очки

### 9.2. Главная идея
Быстро перенести backbone и ЦУ в более чистый сектор.

### 9.3. Правила
- stop low-EV fights
- строить коридор с безопасными adjacent anchors
- переносить ЦУ агрессивнее обычного
- использовать диверсии только чтобы открыть путь/сбить преследователя
- после респавна первые 20–30 ходов — это rebuild tempo, а не месть

### 9.4. После респавна
Так как апгрейды сохраняются, recovery может быть быстрым.
Приоритеты:
1. первая adjacent-опора для ЦУ
2. вторая полоса / backup-link
3. safe HQ route
4. только потом дальние авантюры

## 10) Апгрейды

Здесь есть одна критическая неопределённость: `repair_power` в документации и в API описан неоднозначно. В тексте есть намёк, что он влияет и на стройку; это надо проверить на тестах.

### 10.1. Если `repair_power` действительно бустит стройку
Тогда это S-tier апгрейд.

Рекомендуемый default-путь:
1. `repair_power`
2. `signal_range`
3. `repair_power`
4. `settlement_limit`
5. `earthquake_mitigation`
6. `repair_power`
7. `settlement_limit`
8. `max_hp`
9. `signal_range`
10. дальше по состоянию: `settlement_limit` / `max_hp` / `earthquake_mitigation` / `beaver_damage_mitigation`

### 10.2. Если `repair_power` НЕ бустит стройку
Тогда приоритет смещается:
1. `signal_range`
2. `settlement_limit`
3. `earthquake_mitigation`
4. `max_hp`
5. `signal_range`
6. `settlement_limit`
7. `max_hp`
8. `earthquake_mitigation`
9. дальше situational

### 10.3. Общие правила выбора апгрейда
- `settlement_limit`: когда упираемся в cap или скоро упрёмся
- `signal_range`: когда растёт idle backline и трудно дотягиваться до хороших output
- `earthquake_mitigation`: если теряем темп из-за quakes на стройках
- `max_hp`: если много боёв/рейдов
- `beaver_damage_mitigation`: если реально фармим бобров как часть core-plan
- `vision_range`: если карта разреженная и не хватает информации
- `decay_mitigation`: низкий приоритет, если бот дисциплинированно не бросает стройки

### 10.4. Очень важный нюанс
`max_hp` **не бафает стройки**, а quake/beaver mitigation — бафают и стройки тоже. Поэтому в build-heavy игре `earthquake_mitigation` часто ценнее, чем кажется на бумаге.

## 11) Ручные оверрайды: не ручное управление, а веса поверх алгоритма

Никакого “кликни юнита влево”. Тики быстрые. Нужны только мягкие директивы.

### 11.1. Рекомендуемый формат
```json
{
  "ttl": 30,
  "modeBias": {"econ": 0.0, "aggro": 0.25, "defense": -0.1},
  "directionBias": {"to": [420, 280], "weight": 0.35},
  "sectorBlocklist": [[[390, 260], [430, 300]]],
  "contestCells": [[413, 287]],
  "hqSafetyMargin": 18,
  "beaverBias": 0.4,
  "outputLoadCap": 3,
  "constructionCap": 4,
  "disableBeaverHunts": false
}
```

### 11.2. Что полезно иметь
- `modeBias`
- `directionBias` / waypoint
- `sectorWhitelist` / `sectorBlocklist`
- `contestCells`
- `hqSafetyMargin`
- `beaverBias`
- `riskCap`
- `outputLoadCap`
- `constructionCap`
- `reserveNearHQ`
- `varianceMode` (safe / balanced / high-variance)

### 11.3. Принцип
Любой ручной override должен:
- иметь TTL,
- быть additive/multiplicative к весам,
- не ломать базовые safety-rules (спасение ЦУ, валидность пути и т.д.).

## 12) Fallback и anti-stuck логика

Это обязательно. Иначе бот начнёт молиться на один и тот же невыполнимый `path`.

### 12.1. Общий принцип
У каждого действия хранить:
- тип задачи
- author/output/target
- turn_issued
- expected_effect
- fail_streak

На следующем ходу сравнивать ожидание с миром.

### 12.2. Для стройки
Считать провалом, если:
- нет прироста progress 2 хода подряд
- цель стала недоступной/бессмысленной
- contest проигрывается по tempo
- стройка попала под cap-ловушку и грозит снести старую плантацию/ЦУ

Fallback:
- снять задачу
- blacklist `(target,type,output)` на N ходов
- вернуть автора в общий пул
- если клетка всё ещё стратегична — пересчитать как burst-finish или contest

### 12.3. Для ремонта
Провал/сброс, если:
- цель исчезла
- цель больше не critical
- ремонт уже не меняет survive-probability

### 12.4. Для диверсии
Сброс, если:
- цель не видна 2 хода
- цель вероятно иммунна
- expected enemy repair >= наш damage
- kill ETA вырос за допустимый горизонт

### 12.5. Для бобра
Сброс, если:
- secure_prob last-hit упал ниже порога
- чистый урон <= 5
- kill ETA слишком длинный
- рядом открылась более выгодная contested/defense задача

### 12.6. Универсальное правило против тупого спама
**Один и тот же exact path можно слать не более 2 ходов подряд без подтверждённого позитивного эффекта.**

### 12.7. Filler action
Так как пустая команда даёт ошибку, всегда нужен fallback:
- лучший безопасный build
- или critical repair
- или upgrade
- или relocateMain

## 13) Endgame overlay

С приближением конца раунда вес будущей инфраструктуры падает.

### 13.1. За ~100 ходов до конца
- ниже ценность дальних стройкок без быстрого выхода в очки
- выше ценность secure beaver / quick sabotage / immediate completion

### 13.2. За ~50 ходов до конца
- новые глубокие экспансии только если completion ETA очень короткий
- contested kills и secure bursts становятся заметно ценнее
- repair only if node точно принесёт ещё несколько scoring-turns или держит сеть

### 13.3. Если есть live-статистика по раунду
- при лидерстве — понижать variance
- при отставании — повышать variance и брать более рискованные deny/beaver plays

## 14) Минимальный каркас выбора режима

```python
from enum import Enum

class Mode(Enum):
    ECON = "econ"
    RAID = "raid"
    DEFENSE = "defense"
    CONTESTED = "contested"
    REBASE = "rebase"


def choose_mode(s):
    if s.hq_risk > 0.85 or s.bridge_risk > 0.80:
        return Mode.DEFENSE

    if s.need_rebase or (s.space_quality < 0.35 and s.enemy_pressure > 0.55):
        return Mode.REBASE

    if s.contested_zone_ev > s.free_expand_ev * 0.9 and s.enemy_local_strength >= s.our_local_strength * 0.8:
        return Mode.CONTESTED

    if s.raid_ev > s.expand_ev * 1.15 and s.local_superiority > 1.2:
        return Mode.RAID

    return Mode.ECON
```

## 15) Минимальный каркас аллокатора

```python
from collections import defaultdict


def assign_actions(state, tasks):
    used_authors = set()
    output_load = defaultdict(int)
    commands = []

    # tasks already sorted by descending utility
    for task in tasks:
        while task.need > 0:
            best = best_author_output_pair(state, task, used_authors, output_load)
            if not best:
                break
            author, output, target, base_stat = best
            eff = max(0, base_stat - output_load[tuple(output)])
            if eff <= 0:
                break
            commands.append({"path": [author, output, target]})
            used_authors.add(tuple(author))
            output_load[tuple(output)] += 1
            task.need -= eff

    return commands
```

## 16) Что обязательно проверить в тестовых раундах

Это не “можно бы”, это надо проверить, потому что эти вещи меняют мету.

1. `repair_power` увеличивает ли именно **construction speed**?
2. Бонус за убийство диверсией — это полная ценность клетки или остаточная/иная формула?
3. `immunityUntilTurn` inclusive или exclusive?
4. Видны ли чужие стройки в `construction[]` так же, как свои, и как именно они себя ведут в fog?
5. В ход с `earthquake turnsUntil=0` достроенная в build-phase плантация уже иммунна к quake или нет?
6. Можно ли безопасно переносить ЦУ в тот ход, когда текущая клетка добивается до 100%?
7. Как exactly считается limit с lost plantations и constructions?
8. Что происходит, если лимит превышен и oldest-tie включает ЦУ — реально ли его снести таким образом? (только в безопасном тесте)
9. Как сервер ведёт себя, если path формально валиден, но target в этот ход исчез?
10. Можно ли по логам надёжно восстанавливать причины исчезновения юнитов и использовать это для self-debug?

## 17) Конечная доктрина в одной фразе

**Базово играем в высокотемповую, компактную, не хрупкую экономику; временно переключаемся в рейд/деф/contested/rebase только когда локальная ситуация даёт EV выше, чем очередная волна безопасной экспансии.**

И ещё короче:
- не змейка, а живая лента;
- не честная война, а короткое ограбление;
- не культ 7x7, а культ tempo;
- не ручное микроменеджмент-дерьмо, а режимы + веса + fail-safe.
