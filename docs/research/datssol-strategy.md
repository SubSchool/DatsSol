# DatsSol Strategy Notes

## Что выглядит самым эффективным на старте

### 1. Безопасная связная экспансия важнее ранней агрессии

- Очки за терраформацию приходят каждый ход.
- Потеря ЦУ разрушает всю сеть и режет текущие очки на 5%.
- Из этого следует, что ранняя игра должна максимизировать число активных connected plantations, а не размены с врагом.

### 2. `repair_power` почти всегда лучший первый апгрейд

- По правилам апгрейд влияет не только на ремонт, но и на стройку.
- Это самый прямой буст темпа: быстрее строим backbone, быстрее спасаем ветки после quake и бобров.
- В коде это зашито как первая рекомендация по умолчанию.

### 3. ЦУ нельзя оставлять на почти завершённой клетке

- После 100% терраформации плантация исчезает.
- Если исчезает ЦУ, сеть схлопывается и команда фактически переспавнивается.
- Значит перенос ЦУ должен быть регулярной автоматической процедурой, а не ручной реакцией.

### 4. Boosted-клетки нужны, но только внутри устойчивой решётки

- Клетки с `x % 7 == 0` и `y % 7 == 0` дают больше очков.
- Однако isolated plantations не приносят очков, даже если продолжают терраформировать.
- Поэтому boosted frontier нужно брать только через кардинально связную опорную сеть.

## Дефолтный план по фазам раунда

### Early game

- Строим 2-3 фронтирных узла одновременно, но только там, где сохраняется кардинальная цепочка к ЦУ.
- Быстро закрываем ближайшие boosted-клетки.
- Избегаем строительства сверх лимита без крайней необходимости.

### Mid game

- Увеличиваем количество выходных точек, чтобы не терять слишком много мощности на shared exits.
- Начинаем упреждающе чинить backbone.
- Смотрим на логова бобров, которые режут лучший путь экспансии.

### Late game

- Повышаем роль survival: меньше дальних веток, больше ремонта и relocation ЦУ.
- Саботаж включаем только если он ломает ключевую линию соперника прямо сейчас.
- При дефиците лимита переходим к более плотной сети и точечным boosted-клеткам.

## Приоритеты апгрейдов

1. `repair_power`
2. `settlement_limit`, если headroom <= 2
3. `signal_range`, когда сеть уже разрослась и хочется работать через удобные exits
4. `earthquake_mitigation`, если quake imminent
5. `beaver_damage_mitigation`, если логова регулярно достают backbone
6. `vision_range`
7. `decay_mitigation`
8. `max_hp`

## Когда выгодно бить бобров

- Логово стоит рядом с самой доходной веткой.
- Логово блокирует путь к boosted frontier.
- Есть 2+ connected plantations, способные фокусить цель без развала стройки.
- Убийство логова даёт хороший swing по очкам и освобождает клетку.

Если ни один из этих пунктов не выполняется, обычно выгоднее строить и терраформировать дальше.

## Когда выгодны диверсии

- Враг занял boosted-клетку рядом с нашим фронтиром.
- У нас локальный перевес по числу connected plantations рядом.
- Вражеская ветка висит на одном-двух backbone узлах.
- Саботаж не ломает собственный темп стройки в этот же ход.

## Что измерять с первых тестовых раундов

- окупаемость первого и второго `repair_power`;
- среднее число active connected plantations по ходу раунда;
- частоту потери ЦУ;
- долю ходов, где shared exit режет полезную мощность до 0;
- payoff от бобров относительно чистой экспансии;
- payoff от диверсий по сравнению с ростом собственной сети.

## Разбиение алгоритма по шагам

### `observe`

- получить `arena`;
- получить `logs`;
- сохранить raw state.

### `analyze`

- пересчитать connected component от ЦУ;
- определить frontier кандидатов;
- оценить угрозы: бобры, враг, quake, storm;
- оценить pressure по лимиту поселений;
- оценить риск для текущего ЦУ.

### `decide`

- сформировать intents: repair, beaver attack, sabotage, build, relocate main, upgrade;
- каждому intent дать priority, reason и preferred authors.

### `execute`

- выбрать авторов без конфликта по одному действию на плантацию;
- подобрать exit points в рамках `SR`;
- посчитать реальную мощность после штрафа на повторное использование exit;
- собрать финальный payload.

### `submit`

- отправить `dry-run` или live;
- записать ответ и возможные ошибки;
- обновить turn log и telemetry.
