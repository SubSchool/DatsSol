# DatsBlack 2024 Notes

Источник контракта: [datsblack.openapi.json](../contracts/datsblack.openapi.json)

## Что видно по OpenAPI

- Есть отдельная карта поля: `GET /api/map`.
- Бой идёт по тикам: `tick` возвращается в `scan`, `longScan` и `shipCommand`.
- Цикл управления выглядит как `scan -> decision -> shipCommand`.
- Поддерживаются как минимум два режима регистрации:
  - `deathMatch`
  - `royalBattle`
- Команды отправляются батчем на массив кораблей.
- У корабля есть:
  - позиция, скорость и направление;
  - ограничения на ускорение;
  - радиус сканирования;
  - радиус выстрела и cooldown;
  - статистика успешных попаданий.

## Архитектурные выводы

- Нужен стабильный runtime, который умеет обрабатывать шаги строго по тикам.
- Решение должно быть отделено от транспорта: реальный игровой адаптер должен быть заменяемым.
- Визуализация должна поддерживать:
  - поле большого размера;
  - свои и вражеские юниты;
  - зоны и радиусы;
  - ручные команды и приоритеты поверх автопилота.
- Все действия нужно писать в структурированные логи:
  - какой тик был получен;
  - какая стратегия выбрана;
  - какие команды отправлены;
  - что было переопределено вручную;
  - ошибки API и таймауты.

## Что уже подготовлено в кодовой базе

- `provider`-слой для замены моков на реальную интеграцию;
- runtime с ручным стартом, стопом, рестартом и одиночным тиком;
- базовые стратегии `aggressive`, `neutral`, `pacifist`;
- отдельный раздел логов с фильтрами и экспортом;
- моковая космическая карта для отладки визуализации до получения задания.
